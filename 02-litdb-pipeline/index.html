LitDB 專利分析管線VC-2026-02 · 概念一

從「靜態資料庫」改造成「一句話輸入的專利分析管線」

導讀：這個構想在解決什麼問題（白話版）

主題目的

現在的 LitDB 花很多時間維護，但沒辦法自動化，資料整理得亂，介面也只有 CJ 自己看得懂。這個構想要解決的問題只有一個：讓「輸入一句話（器材名稱）」就能自動跑出這個器材的專利地景報告，不再需要人工一件一件讀專利、手填資料。

脈絡（現在為什麼卡住）

根本原因不是 schema 設計得不好，是三件本來該分開的事被綁在一起：

1

資料取得——現在是人工讀專利、手動填欄位

2

資料整理——schema 隨著每次新需求手動長大，欄位越加越多、越加越亂

3

資料呈現——介面跟資料綁死，所以只有建立者自己知道怎麼用

只要第 1 步還是人工，第 2、3 步就永遠追不上、永遠會亂。所以重寫的重點不是換一個更漂亮的 JSON schema，而是把「取得」自動化，讓它變成一條會自己跑的管線。

方法（新架構怎麼運作）

未來使用情境：CJ 打一句話，例如「同軸切片針」，管線自動跑完以下流程，最後給一份報告。

管線流程圖

介面只需做兩件事：「輸入一句話」和「看報告」。①～⑥全部自動跑，人不用再逐件讀專利、手填欄位——現在花在 LitDB 上的時間，本質上就是在人肉執行這六步。

各階段白話對照表

階段現在（人工）未來（自動）
① 概念展開CJ 自己想同義詞、猜分類碼AI 一次列出同義詞、上下位概念、CPC 碼
② 多源檢索CJ 自己上不同網站查、複製貼上四個 API 平行查詢，結果自動合併去重
③ 引用滾雪球目前完全沒做，因為太花時間自動追引用鏈，直到沒有新專利出現為止
④ 法律狀態過濾CJ 手動查每件專利是否過期API 直接回傳法律狀態，自動分類
⑤ Claims 拆解CJ 讀請求項、手動理解限制範圍AI 逐項拆解成結構化清單
⑥ 地景綜合不存在——目前沒有能力做跨專利比對自動產出「這個設計方向被誰卡住」的矩陣

新舊架構對照

LitDB v1（現況）

  • 資料取得、整理、呈現綁在一起
  • schema 隨需求手動長大，欄位混亂
  • 介面跟資料耦合，只有 CJ 會用
  • 沒有滾雪球，人工挑漏
  • 過期與垃圾專利沒有區分機制

LitDB v2（管線構想）

  • 取得（API）／整理（AI）／呈現（報告）三層分離
  • 資料庫只是管線的中間產物，不是終點
  • 介面只做「輸入一句話＋看報告」
  • 引用滾雪球自動跑到收斂
  • 垃圾專利評估另立概念（VC-2026-03 待補）
