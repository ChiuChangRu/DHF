DHF 本體 ER 模型v2.0

部署路線：Cloudflare Email Routing + Worker + D1（測試/中繼層）→ 鼎新 ERP（正式系統）· 依 21 CFR 820.30 / ISO 13485 §7.3 / ISO 14971

導讀：這份文件在做什麼（白話版）

主題目的

你現在用 email 收案件執行人寄來的 DHF 文件；未來這些資料要能無痛搬到鼎新系統。這份 ER 模型就是先把「資料要長什麼樣子」設計好，讓現在（Cloudflare D1）跟未來（鼎新）用同一套邏輯，搬家時不用重新整理資料。

脈絡（整個故事怎麼跑）

1

執行人寄一封 email，附上一份文件（例如驗證報告）

2

系統收到信，AI 讀一下，猜這份文件屬於 DHF 哪個章節（設計輸入？驗證？）、有沒有風險要標記

3

這些猜測先存起來但不算數，要等執行人自己確認過才正式生效——刻意這樣設計，是因為法規稽核不能讓 AI 自己下判斷、沒有人簽核

4

之後鼎新系統上線，這些資料要整批搬過去（見下方「鼎新遷移對應」白話表）

方法（下方各節在做什麼）

章節白話說明
一、ER 圖把上面故事裡出現的「案件」「人」「文件」「章節」「風險」畫成方框和連線，方框是「一種資料」，連線是「這兩種資料怎麼互相關聯」
二、關聯一覽把 ER 圖裡的連線用文字表格重講一次，寫清楚「誰對誰、一個對多個還是多個對多個」
三、D1 建表腳本照著 ER 圖直接寫成資料庫可以執行的程式碼，現階段測試部署用
四、鼎新遷移對應逐一表格說明「這份資料未來搬到鼎新，會變成什麼、要先確認什麼」——最容易看不懂的通常是這節，下方另外用白話版重講一次
五、Mermaid 原始碼同一張 ER 圖的純文字版，可貼到 GitHub、Notion 等工具重新畫出來，方便日後交接或簡報

「四、鼎新遷移對應」白話重講

大部分表格搬家很單純（案件、人員、章節碼表），真正要花心思確認的只有兩個地方，下表用粗體橘字標出。

現在（D1）未來搬去哪白話備註
案件（PROJECT）專案管理／研發工單用案件編號互相對照即可，簡單
人員（PERSON）人事主檔把 email 換成鼎新的員工編號，建一次對照表就好
文件（DOCUMENT）文管模組檔案本身從 GitHub 搬過去，資料欄位批次匯入；檔案的「指紋」(hash) 搬完要再驗一次確保沒壞掉
章節碼表（DHF_SECTION）文管分類代碼固定的九大分類，鼎新那邊建同樣代碼即可，很單純
文件×章節（DOC_SECTION_LINK）文件分類關聯要先確認鼎新是否支援「一份文件同時掛多個分類」，如果只能選一個，這裡會卡住
風險評估（RISK_ASSESSMENT）品質模組或自訂表單鼎新可能沒有專門對應 ISO 14971 的欄位，可能要用自訂表單接住
追溯關聯（TRACE_LINK）通常沒有原生對應也要先確認——這是「設計輸入有沒有被驗證到」這種查詢的核心，鼎新如果沒有，可能得繼續留在 D1 或做成報表附件
稽核紀錄（AUDIT_LOG）鼎新系統日誌舊的 AI 判斷紀錄要匯出封存，確保稽核時查得到「當初 AI 時期發生了什麼」

一、實體關聯圖（Crow's Foot 簡化記法）

核心實體 主檔 / 參照表 多對多關聯表 稽核（Append-only） 實線＝外鍵關聯　虛線＝多型參照
建模三原則：①「文件屬於哪個 DHF 章節」是多對多，用 DOC_SECTION_LINK 關聯表承載，而非在 DOCUMENT 上放單一分類欄位——與 LitDB 專利滾雪球 Citation 表是同一個 ER 思維。②AI 的判斷（分類、風險）一律以 ai_suggested / assigned_by='AI' / status='ai_draft' 標記，經人工確認才生效。③每個核心實體都預留 erp_ref 欄位，為鼎新遷移建立雙向對照。

二、關聯一覽

關聯基數語意
PROJECT → DOCUMENT1 : N每份文件必隸屬一個案件（如 TR-RD2-26-020、Kaneko 同軸切片針專案）
PERSON → DOCUMENT1 : N提交者；收信 Worker 以 PERSON.email 白名單驗證寄件人
DOCUMENT ↔ DHF_SECTIONN : M經 DOC_SECTION_LINK；一份驗證報告可同屬 Design Verification 與回應某 Design Input
DOCUMENT → RISK_ASSESSMENT1 : N一份文件可觸發多筆風險評估（獨立實體，非 Document 內嵌欄位）
DOCUMENT → TRACE_LINK（×2）1 : N自我參照關聯表；支撐 Design Input → Output → V&V 的追溯矩陣查詢
PERSON → RISK_ASSESSMENT1 : N風險評估確認者（confirmed_by），稽核時的簽核依據
AUDIT_LOG → 各實體多型entity_type + entity_id 參照任何實體；僅新增、不可修改

三、Cloudflare D1 建表腳本（SQLite 方言）

可直接以 wrangler d1 execute dhf-db --file=schema.sql 部署。D1 定位為測試與中繼層，正式紀錄以遷移後的鼎新系統為準。

-- DHF Schema v2.0 · Cloudflare D1 (SQLite) · 2026-07-02
PRAGMA foreign_keys = ON;

CREATE TABLE project (
  project_id      INTEGER PRIMARY KEY AUTOINCREMENT,
  project_code    TEXT NOT NULL UNIQUE,          -- 例：TR-RD2-26-020
  name            TEXT NOT NULL,
  device_type     TEXT,                          -- HD catheter / biopsy needle...
  regulatory_path TEXT CHECK(regulatory_path IN ('510k','TFDA','both')),
  status          TEXT DEFAULT 'active',
  erp_ref         TEXT,                          -- 鼎新專案/工單號（遷移對照）
  created_at      TEXT DEFAULT (datetime('now'))
);

CREATE TABLE person (
  person_id   INTEGER PRIMARY KEY AUTOINCREMENT,
  name        TEXT NOT NULL,
  email       TEXT NOT NULL UNIQUE,              -- 收信白名單驗證依據
  department  TEXT,
  role        TEXT,
  is_active   INTEGER DEFAULT 1,
  erp_ref     TEXT                               -- 鼎新員工編號
);

CREATE TABLE dhf_section (
  section_id   INTEGER PRIMARY KEY,
  section_code TEXT NOT NULL UNIQUE,             -- DP/DI/DO/DR/DV/DVal/DT/DC/RMF
  name_zh      TEXT NOT NULL,
  name_en      TEXT NOT NULL,
  cfr_ref      TEXT,                             -- 21 CFR 820.30(x)
  iso_ref      TEXT                              -- ISO 13485 7.3.x
);

CREATE TABLE document (
  doc_id            INTEGER PRIMARY KEY AUTOINCREMENT,
  project_id        INTEGER NOT NULL REFERENCES project(project_id),
  submitter_id      INTEGER NOT NULL REFERENCES person(person_id),
  filename          TEXT NOT NULL,
  version           TEXT DEFAULT '1.0',
  doc_type          TEXT,
  file_hash         TEXT NOT NULL UNIQUE,        -- SHA-256，防重複與竄改比對
  file_url          TEXT NOT NULL,               -- GitHub repo 路徑
  email_message_id  TEXT,                        -- 原始信件 Message-ID，溯源
  ai_classification_json TEXT,                   -- Claude 分類原始輸出
  confidence_tier   TEXT CHECK(confidence_tier IN ('green','yellow','red')),
  status            TEXT DEFAULT 'pending_review'
                    CHECK(status IN ('pending_review','confirmed','rejected')),
  confirmed_by      INTEGER REFERENCES person(person_id),
  confirmed_at      TEXT,
  erp_ref           TEXT,                        -- 鼎新文管單號
  received_at       TEXT DEFAULT (datetime('now'))
);

CREATE TABLE doc_section_link (
  link_id       INTEGER PRIMARY KEY AUTOINCREMENT,
  doc_id        INTEGER NOT NULL REFERENCES document(doc_id),
  section_id    INTEGER NOT NULL REFERENCES dhf_section(section_id),
  assigned_by   TEXT NOT NULL CHECK(assigned_by IN ('AI','HUMAN')),
  ai_confidence REAL,                            -- 0–1，對應三色信心分級
  is_confirmed  INTEGER DEFAULT 0,
  confirmed_by  INTEGER REFERENCES person(person_id),
  UNIQUE(doc_id, section_id)
);

CREATE TABLE risk_assessment (
  risk_id         INTEGER PRIMARY KEY AUTOINCREMENT,
  doc_id          INTEGER NOT NULL REFERENCES document(doc_id),
  project_id      INTEGER NOT NULL REFERENCES project(project_id),
  hazard          TEXT NOT NULL,
  harm            TEXT,
  severity        INTEGER,
  probability     INTEGER,
  risk_level      TEXT,
  mitigation      TEXT,
  iso14971_clause TEXT,
  ai_suggested    INTEGER DEFAULT 1,             -- 1=AI 建議，0=人工建立
  status          TEXT DEFAULT 'ai_draft'
                  CHECK(status IN ('ai_draft','confirmed','rejected')),
  confirmed_by    INTEGER REFERENCES person(person_id),
  confirmed_at    TEXT,
  created_at      TEXT DEFAULT (datetime('now'))
);

CREATE TABLE trace_link (
  trace_id    INTEGER PRIMARY KEY AUTOINCREMENT,
  from_doc_id INTEGER NOT NULL REFERENCES document(doc_id),
  to_doc_id   INTEGER NOT NULL REFERENCES document(doc_id),
  link_type   TEXT NOT NULL CHECK(link_type IN
              ('input_to_output','output_to_verification',
               'output_to_validation','risk_to_mitigation','supersedes')),
  created_by  INTEGER REFERENCES person(person_id),
  created_at  TEXT DEFAULT (datetime('now'))
);

CREATE TABLE audit_log (
  log_id      INTEGER PRIMARY KEY AUTOINCREMENT,
  entity_type TEXT NOT NULL,                     -- 'document' / 'risk_assessment'...
  entity_id   INTEGER NOT NULL,
  action      TEXT NOT NULL,                     -- created / classified / confirmed...
  actor_type  TEXT NOT NULL CHECK(actor_type IN ('AI','HUMAN','SYSTEM')),
  actor_id    INTEGER,
  detail_json TEXT,
  created_at  TEXT DEFAULT (datetime('now'))
);
-- audit_log 僅允許 INSERT；應用層禁用 UPDATE/DELETE

CREATE INDEX idx_doc_project  ON document(project_id);
CREATE INDEX idx_doc_status   ON document(status);
CREATE INDEX idx_dsl_section  ON doc_section_link(section_id);
CREATE INDEX idx_risk_status  ON risk_assessment(status);
CREATE INDEX idx_audit_entity ON audit_log(entity_type, entity_id);

四、鼎新（Digiwin）遷移對應規劃

D1 作為中繼層的核心價值：所有實體已是正規化 SQL 表格，遷移時只做「欄位對應 + 批次匯出」，不需重新整理資料。

D1 實體鼎新對應模組（暫定）遷移備註
PROJECT專案管理 / 研發工單以 project_code 為對照鍵，回填 erp_ref
PERSON人事主檔（員工編號）email → 員工編號對照表，一次性建立
DOCUMENT文件管理（文管）模組檔案本體自 GitHub 匯出上傳；metadata 走批次匯入；file_hash 遷移後重驗
DHF_SECTION文管分類代碼主檔種子資料，於鼎新端建立相同代碼即可
DOC_SECTION_LINK文件分類關聯若鼎新僅支援單一分類，需確認是否支援多分類標籤；此為遷移前必查項
RISK_ASSESSMENT品質/風險模組（或自訂表單）ISO 14971 欄位若無對應模組，評估以鼎新自訂電子表單承接
TRACE_LINK—（多半無原生對應）追溯矩陣可能需保留於 D1 或匯出為報表附件；此為遷移前必查項
AUDIT_LOG鼎新系統日誌 + 保留原始 D1 匯出檔遷移前的 AI 分類軌跡以唯讀匯出封存（CSV/JSON），確保稽核可追溯至 email agent 時期

遷移三步驟

階段內容
①凍結停止 email agent 寫入 D1（或切為唯讀鏡像模式），匯出全部八表為 CSV + 完整 audit_log
②對應與鼎新導入顧問確認兩個必查項（多分類支援、追溯關聯承接方式），完成欄位對應表並試匯入一個小專案驗證
③切換批次匯入 → 回填各表 erp_ref → 抽樣核對 file_hash → email agent 改為寫入鼎新 API（或維持 D1 前置、定期同步）
法規提醒：遷移本身屬於 QMS 下的系統變更，建議以一份簡要的資料遷移驗證紀錄（匯出筆數 = 匯入筆數、hash 抽核、追溯關聯抽核）納入 DHF/QMS 文件，避免稽核時被問「AI agent 時期的紀錄去哪了」。
